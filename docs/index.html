Spatial data has a more complicate ecosystem

Spatial data provides information about units of analysis, just like "flat" data. The difference is the need to provide a spatial referent.

  • With dealing with time, one must provide a coordinating reference (e.g. time zone) so that all parties are aligned.
  • Dealing with space is the same idea, but in practice it is more complicated.
  • The world has converged on common increments of time.
  • Time generally proceeds in one direction (insofar as we experience it).

Spatial reference systems help us stay coordinated

A Spatial Reference System , at its core, consists of a datum and a projection. These are both needed, not only because of variance in measurement systems, but also because of the earth's irregular shape.

A datum is similar to other CRSs

The point is to establish a reference point and an unambiguous "address" for any point in the system related to that point.

Projections are about consistent measurement

Projections are needed to map the surface of a spheroid to an imperfect representation (usually a 2-D plane). All projections are wrong, but some are useful.

Shapefiles are the dominant format

Shapefiles were developed by ESRI, and remain the most widely available format for spatial data. While you may often see more, there are only three required components:

  • shp: contain the actual coordinates of feautures/shapes
  • dbf: contain data associated with each feature
  • shx: contain an index info to facilitate references of individual features
  • xml (bonus): tend to contain meta-information about shapefile data (inclusive of origin and CRS)

GeoJSON files are necessarily similar

They contain the same spatial reference data, but all in the same file. Moreover, they are entirely comprised of text. Both attributes make them easier to programmatically manipulate, and much easier to deal with on the fly (see Github).
